# NewsControl

> 최종 수정: 2026-07-12

보도기술팀 업무 관리 시스템. Express 서버(`server/`)와 Vite + React 클라이언트(`client/`)로 구성된 모노레포입니다.

## 문서 모음

```bash
npm run doc
```

후 `http://localhost:4400/doc/index.html` 접속 — README·PRD·[개발 기록(DEVLOG)](doc/DEVLOG.md)을 검색 가능한 화면으로 볼 수 있습니다. 개발 흐름·문제·해결 방법은 기능이 바뀔 때마다 `doc/DEVLOG.md`에 이어서 기록합니다.

## 작업 시작 전 점검

```bash
npm run precheck
```

Git 상태, Railway 프로덕션 배포/데이터 상태, (레거시) Vercel 상태, 저장 경로 코드 정합성을 읽기 전용으로 확인합니다(배포 없음). 특히 며칠 이상 간격을 두고 다시 작업할 때 실행 — 이 프로젝트는 배포가 git과 무관하게 이뤄질 수 있어서(`doc/DEVLOG.md` 교훈 #9, #11) "지난번 상태 그대로겠지"라는 가정이 위험합니다. AI 에이전트(Cursor/Claude Code)용 지침은 `AGENTS.md` 참고.

## 실행방법

### 1. 의존성 설치 (최초 1회)

```bash
npm run install-all
```

루트, `server/`, `client/`의 의존성을 한 번에 설치합니다.

### 2. 개발 서버 실행

```bash
npm run dev
```

`server`(Express, nodemon)와 `client`(Vite)를 동시에 실행합니다.

- 서버: http://localhost:5001
- 클라이언트: http://localhost:5173

브라우저에서 http://localhost:5173 접속하면 됩니다. `BLOB_READ_WRITE_TOKEN` 환경변수가 있으면(플랫폼 무관) Vercel Blob에, 없으면 `RAILWAY_VOLUME_MOUNT_PATH`(Railway Volume 연결 시 자동 주입)에, 그마저 없으면 `data/` 폴더의 JSON 파일에 데이터와 업로드 파일을 저장합니다.

## 배포

### Vercel

`vercel.json` 설정 그대로 `client`는 정적 빌드, `server`는 서버리스 함수로 배포됩니다. 서버리스 환경은 로컬 디스크가 휘발되므로 `BLOB_READ_WRITE_TOKEN`(Storage 탭에서 Blob 스토어 생성) 설정이 필수입니다.

### Railway

`railway.json`이 빌드(`npm run build`)와 시작(`npm start`) 명령을 지정합니다. 서버 하나가 API와 빌드된 클라이언트를 함께 서빙하는 상시 실행 컨테이너로 동작합니다.

1. Railway 프로젝트를 생성/연결
2. 서비스에 **Volume**을 추가(아무 마운트 경로나 지정, 예: `/data`) — Railway가 `RAILWAY_VOLUME_MOUNT_PATH`를 자동으로 주입하며, 서버는 이 경로에 데이터/업로드 파일을 저장한다. Vercel Blob 용량이나 요금과 무관하게 Railway 자체 디스크만 사용
3. `railway up` (또는 GitHub 연동 후 자동 배포)

> `BLOB_READ_WRITE_TOKEN`을 환경변수로 넣으면 Railway에서도 Volume 대신 Vercel Blob을 계속 쓸 수 있지만, Blob 무료 한도를 다 쓴 상태라면 Volume 방식을 쓰는 편이 추가 비용이 없다.

### 3. 프로덕션 빌드 (선택)

```bash
npm run build --prefix client
```

빌드 결과물(`client/dist`)은 로컬 모드에서 서버가 정적 파일로 함께 서빙합니다.

## 전광판(`?display`) 표시 기준

`http://localhost:5173/?display` (또는 대시보드의 "전광판" 버튼)로 여는 전광판 화면은 두 영역으로 구성됩니다. 관련 코드: `client/src/App.tsx`의 `isDisplayMode` 블록.

### 1. 상단 한줄공지 바 (1분마다 다음 줄로 스크롤)
- 조건: `announcement.expiresAt > 현재시각` (아직 만료 안 됨)
- 활성 한줄공지가 여러 개면 한 번에 한 줄만 표시하고, 1분마다 다음 공지로 자동 전환됨(마지막이면 처음으로 순환) — `Math.floor(현재시각 / 60000) % 활성개수`로 현재 몇 분인지에 따라 표시할 공지를 결정
- 전환 시 아래에서 위로 슬라이드-인 되는 스크롤 애니메이션 적용
- `expiresAt`은 등록 시 고른 기간(3일 / 1주)만큼 현재시각에 더해서 결정됨
- 중요 표시(등록 시 체크박스 또는 목록의 📌 버튼)된 한줄공지는 글자 전체가 빨간색으로 강조됨

### 2. 메인 카드 그리드 (표시 개수에 따라 자동으로 1개/2개/3개 이상 레이아웃 전환)
후보 조건 — `isNoticeLiveForDisplay()`:
1. `showFrom`(예약 공개 시각)이 설정돼 있고 아직 도래하지 않았으면 제외
2. **공지일(1일/3일/7일)을 선택하지 않은 공지(기본)**: 예정 시각 기준 **3시간 전부터 30분 후까지**만 후보 — 좁은 창을 지나면 후보에서 제외되어 화면에서 사라짐
3. **공지일을 선택한 공지**: 예정 시각과 무관하게 **등록(또는 마지막 수정)한 순간(`noticeDayFrom`)부터 선택한 일수(1일/3일/7일) 동안** 계속 후보로 유지됨
4. `isPinned`(중요 표시) 여부는 후보 선정에 영향 없음 — 빨간 강조 스타일만 결정

**그리드 레이아웃** — 표시할 공지 개수에 따라 자동 전환 (`data-count` 속성 + CSS 커스텀 프로퍼티로 폰트 크기까지 함께 스케일):
- **1개**: 전체화면 1칸, 가장 큰 폰트
- **2개**: 좌우로 반씩 분할
- **3개 이상**(표시할 공지가 없을 때 포함): 기존처럼 2×2 격자(최대 4개), 기본 폰트 크기

**표시 우선순위(칸이 채워지는 순서)** — `compareNoticeDisplayOrder()`, 왼쪽 위가 1순위·오른쪽 아래가 최하위:
1. 공지일(1일 이상)을 선택한 장기 공지는 **항상 최하위** — 서로간에는 `noticeDayFrom`(등록 시점)이 이른 순
2. 나머지(당일 특정 시각 공지)는 예정 시각이 아직 안 지났으면 **가까운 시각 순**으로 높은 우선순위
3. 예정 시각이 지나 30분 유예 기간만 남아 잠깐 표시 중인 공지는 **2순위**로 밀리고, 유예 기간이 끝나면 사라짐

- 슬롯이 이미 차 있으면 이미 표시 중인 공지는 더 이상 후보가 아니게 될 때까지 슬롯을 유지(sticky)하며, 빈 자리만 위 우선순위로 채움
- 이미지가 첨부된 공지는 텍스트 대신 이미지만 크게 표시
- 관련 코드: `App.tsx`의 `isNoticeLiveForDisplay()`, `compareNoticeDisplayOrder()`, `NOTICE_DURATION_DAYS`, `NOTICE_DEFAULT_WINDOW_BEFORE_MS`/`_AFTER_MS`

**카드 강조 스타일 우선순위**: 깜빡임(빨강, 임박 1회성) > 중요/pinned(빨강, 제목 강조) > 임박 알람(노랑) > 지난 공지(연노랑, "완료" 배지) > 기본 — 어느 상태든 본문 글자 크기 자체는 줄어들지 않고 그리드 구간별 크기를 그대로 따름

### 3. 임박 알람 (진한 노란색)
설정(`설정 > 공지 알림 타이밍`)의 **"노란색 강조 (분 전)"** 값(기본 30분)을 기준으로, 공지 예정 시각까지 남은 시간이 그 값 이하로 들어오면 알람이 시작됩니다.

- 10분 간격으로 1분간만 진한 노란색 배경으로 깜빡임 (예: 기본값 30분 기준 → 예정 시각 30분 전, 20분 전, 10분 전에 각 1분씩 알람)
- 예정 시각이 지나면(0분 이하) 알람은 멈추고 "지난 공지" 스타일로 전환됨
- 관련 코드: `App.tsx`의 `isInAlarmWindow()`, `ALARM_INTERVAL_MINS`(10), `ALARM_DURATION_MINS`(1), CSS는 `App.css`의 `.dm-card.dm-alarm`

## 행사/점검 아카이브

사이드바의 "행사 아카이브"와 "점검 아카이브"는 별도로 관리됩니다.

- **행사 아카이브**(`periodic_events`): 선거 개표방송, 창사 기념 특집 등 주기적 대형 행사. 이름·주기·"핵심 기술 가이드" 목록을 기록.
- **점검 아카이브**(`inspections`): 정기 시설점검 등. 이름·주기·"점검시 유의사항"(과거 점검 후 발견된 이상증상 등) 목록을 기록 — 공지를 등록/수정할 때 "관련 점검"으로 선택하면 유의사항을 미리 보고 공지 내용에 그대로 옮겨 담을 수 있음.
- 두 아카이브 모두 이름·주기·목록 항목을 생성 후에도 자유롭게 수정할 수 있고, 항목마다 자유 서술 "내용"과 이미지/PDF 첨부(Ctrl+V 붙여넣기 지원)를 추가할 수 있음.
- 대시보드의 "사용설명서" 버튼에서 전체 사용법(공지 등록, 표시 규칙, 우선순위, 아카이브 활용법)을 확인할 수 있음.
