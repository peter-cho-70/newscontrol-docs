# DEVLOG — NewsControl 개발 기록

> 최종 수정: 2026-07-22

> 보도기술팀 업무 관리 시스템의 개발 흐름, 마주친 문제와 해결 방법을 시간순으로 기록한다.
> 새 기능을 추가하거나 문제를 해결할 때마다 이 문서 최상단(타임라인 맨 앞)에 새 항목을 이어서 기록한다.
> 항목을 추가/수정할 때마다 위 "최종 수정" 날짜와 해당 항목 제목 옆의 날짜를 함께 갱신한다.
> 보기 좋은 화면으로 읽으려면 `npm run doc` 실행 후 `http://localhost:4400/doc/index.html` → 개발 기록, 또는 배포된 [newscontrol-docs](https://peter-cho-70.github.io/newscontrol-docs/) 사이트에서 바로 확인.

## 프로젝트 개요

- **목적**: 보도기술팀(TD/VD/AD/LD/SD)의 업무 노하우를 데이터베이스화하고, 전광판 공지 시스템으로 스튜디오 현장에 실시간 안내를 제공 (`PRD.md` 참고)
- **구성**: 모노레포 — `client/`(Vite + React + TypeScript), `server/`(Express), `data/`(JSON 시드 데이터)
- **배포**: Vercel(`vercel.json`) 또는 Railway(`railway.json`, 상시 실행 Node 서버가 클라이언트 빌드까지 함께 서빙)
- **데이터 저장**: 우선순위 `BLOB_READ_WRITE_TOKEN`(Vercel Blob, 플랫폼 무관) → `RAILWAY_VOLUME_MOUNT_PATH`(Railway Volume) → `data/initial_work_guide.json`(로컬 파일). 업로드 파일도 동일 우선순위로 저장되어 컨테이너 재배포에도 유실되지 않음

## 타임라인

### 1. 초기 커밋 (2026-06-22, `46f310a`)
Vite+React 프런트엔드와 Express 백엔드로 기본 골격 구성. 역할별 업무 가이드, 프로그램별 주의사항 등 PRD의 Phase 1~2에 해당하는 정적 구조를 구현.

### 2. Vercel 배포 초기 트러블슈팅 (2026-06-22)
로컬에서는 문제없이 돌아가던 앱이 Vercel에 올리자 연쇄적으로 깨졌다. 세 번의 커밋에 걸쳐 원인을 하나씩 제거했다.

| 커밋 | 문제 | 원인 | 해결 |
|---|---|---|---|
| `b15b2f7` | 업로드 라우트 호출 시 모든 요청이 500 에러 | `require('sharp')`를 모듈 최상단에서 호출 → Vercel 서버리스 런타임의 네이티브 바이너리 로딩 문제로 **모듈 로드 자체가 크래시**, 그 라우트를 쓰지 않는 다른 요청까지 전부 죽임 | 업로드 핸들러 **내부에서 지연 로딩(lazy require)** 하고, 실패해도 썸네일 생성만 건너뛰도록 try/catch로 감쌈 |
| `aed0241` | SPA 루트 경로(`/`)가 404 | `vercel.json`의 라우팅이 filesystem-handle 방식이 아니어서 정적 라우트보다 우선순위가 꼬임 | `vercel.json` 라우팅 방식을 filesystem 우선 처리로 변경 |
| `4c48111` | 정적 자산(`/assets/*`, favicon, icons)이 404 | `@vercel/static-build`는 빌드 결과물을 `dist/` 밑이 아니라 **빌드 `src` 디렉터리 경로(`client/`) 그대로** 배포하는데, 번들된 HTML은 루트 상대 경로(`/assets/...`)를 참조 | `vercel.json`에 `/assets/(.*)`, `/favicon.svg`, `/icons.svg`를 `client/...`로 명시적으로 rewrite하는 라우트 추가 |

**교훈**: `@vercel/static-build` + 모노레포 조합에서는 "로컬 빌드 결과물 경로 = 배포 후 서빙 경로"라는 가정이 깨진다. 새로운 정적 자산 경로를 추가할 때마다 이 매핑을 다시 점검해야 한다.

### 3. UI 리디자인 (2026-06-22, `7871c89`)
어두운 남색 SaaS 톤 팔레트를 종이 질감의 따뜻한 색감 + 세이지그린 포인트로 교체. Pretendard 웹폰트 적용으로 한글 타이포그래피 개선. 이때까지는 순수 프런트엔드 스타일 작업이라 배포 이슈는 없었음.

### 4. 공지 데이터 영속성 문제 (2026-06-22~07-02, `769a75b`)
**문제**: 서버리스 환경에서 공지사항을 `/tmp`에 쓰고 있었는데, 서버리스 함수는 콜드스타트/재배포마다 인스턴스가 새로 뜨므로 `/tmp` 내용이 날아간다 → 방금 추가한 공지가 사라지는 현상 발생.

**해결**: 데이터를 **Vercel Blob**에 저장하도록 전환 (`server/index.js`의 `BLOB_KEY = 'newscontrol-guide.json'`). 추가로, blob 조회가 일시적으로 실패한 경우와 "blob이 아직 존재하지 않는 최초 상태"를 구분해서, **일시적 읽기 에러일 때는 시드 데이터로 덮어쓰지 않도록** 처리 — 안 그러면 네트워크 hiccup 한 번에 실제 데이터가 시드 데이터로 롤백되는 사고가 날 수 있었음.

같은 커밋에서 UX 개선도 다수 포함:
- 대시보드에 오늘의 한줄공지 + 전광판 실시간 미리보기 노출
- 전광판 슬롯을 "sticky"하게 유지 — 다가오는 공지가 시간 되기 전에 화면에서 밀려나지 않게
- 공지 카드 텍스트 오버플로우 수정, 높이 제한 + 내부 스크롤
- 리허설처럼 반복되는 일정을 위해 퀵등록에서 날짜/시간 슬롯 최대 3개(10분 단위) 지원

### 5. 위치 프리셋 · 이미지 첨부 · 날짜 버그 (2026-07-02, `a317c52`)
**문제**: 날짜 입력 기본값을 `toISOString()`으로 채웠는데, 이는 **UTC 기준 문자열**이라 한국 새벽 시간대(KST 자정~오전 9시 사이)에는 하루 전 날짜가 표시되는 버그가 있었음.

**해결**: 로컬 타임존 기준으로 날짜를 포맷하는 헬퍼로 모든 날짜 입력 초기화 지점을 교체.

그 외:
- 공지 추가/수정 폼에 위치 프리셋 칩(뉴스센터A/B, 스포츠, 사무실, 기타) — 자유 입력도 유지
- 공지에 이미지 첨부 지원, 전광판에서는 이미지가 있으면 텍스트 카드 대신 이미지만 표시
- 기존 공지 수정 시 날짜/시간도 변경 가능하도록 개선

### 6. 전광판 알람/스크롤/공지일 (2026-07-02, `0a5b835`)
전광판(`?display`) 표시 로직을 상세화:
- **임박 알람**: 설정한 "노란색 강조" 시간(기본 30분 전) 이내로 들어오면, 10분 간격으로 1분씩 진한 노란색으로 깜빡이는 알람 표시
- **한줄공지 스크롤**: 활성 한줄공지가 여러 개면 1분마다 자동으로 다음 줄로 슬라이드 전환
- **공지일(표시 기간)**: 일반 공지에도 1일/3일/7일 표시 기간을 지정할 수 있게 해서, 예정 시각이 지나도 지정 기간 동안은 계속 노출
- 이미지 붙여넣기(Ctrl+V)로도 첨부 가능, 텍스트 없이 이미지만으로도 등록 가능
- README에 전광판 표시 규칙을 상세히 문서화 (`README.md` 참고 — 슬롯 채우기 규칙, 카드 강조 우선순위 등)

### 7. 개발 문서 체계 정리 (2026-07-11)
루트에 있던 `DEVLOG.md`를 `doc/DEVLOG.md`로 옮기고, `autoplay` 프로젝트의 `doc/index.html` + `doc/viewer.html` 패턴을 그대로 가져와 문서 허브를 구성. 외부 라이브러리 없이 동작하는 자체 마크다운 렌더러(표/코드블록/체크박스/TOC 지원)와 라이트/다크 테마 토글을 그대로 재사용. `npm run doc`로 로컬 정적 서버를 띄워 `doc/index.html`에서 README·PRD·DEVLOG를 검색·열람할 수 있게 함.

**문제**: 처음에 `python3 -m http.server 4400 --directory doc`로 doc 폴더 자체를 서버 루트로 띄웠더니, `viewer.html?file=../README.md` 같은 상위 경로 링크가 전부 404. `--directory`로 지정한 폴더가 서버의 절대 루트가 되어 그 바깥(저장소 루트의 README.md·PRD.md)에는 애초에 접근할 수 없었기 때문.
**해결**: 저장소 **루트**를 서빙하도록(`python3 -m http.server 4400`, `--directory` 제거) 바꾸고, 문서 허브는 `http://localhost:4400/doc/index.html`로 접속하는 방식으로 전환.

### 8. NewsControl 문서 사이트를 GitHub Pages로 배포 (2026-07-11)
`mbcnewscontriol` 저장소는 private이라 여기서 바로 Pages를 켤 수 없어서(private repo Pages는 유료 플랜 필요 + 결과물은 어차피 공개), `autoplay`/`mars-master-docs`와 같은 패턴을 그대로 재사용: 문서만 담은 별도의 **public** 저장소를 새로 만들어 그쪽에서 Pages를 켰다.

- 새 저장소 [`peter-cho-70/newscontrol-docs`](https://github.com/peter-cho-70/newscontrol-docs) — `README.md`, `PRD.md`, `doc/`(index.html·viewer.html·DEVLOG.md) 미러 + 루트 `index.html`(→ `doc/index.html` 자동 리다이렉트). GitHub Pages 활성화 → https://peter-cho-70.github.io/newscontrol-docs/
- 여러 프로젝트 문서 사이트를 한곳에서 찾아가도록 개인 문서 허브 저장소 [`peter-cho-70/peter-cho-70.github.io`](https://github.com/peter-cho-70/peter-cho-70.github.io)도 신설 — MARS/Master, NewsControl 두 문서 사이트로 연결되는 카드형 랜딩 페이지. → https://peter-cho-70.github.io/
- 동기화 방식은 **수동**으로 결정: 코드 저장소에서 `doc/DEVLOG.md`·`README.md`·`PRD.md`가 바뀌면 그때그때 `newscontrol-docs`로 복사→커밋→push. 업데이트 빈도가 낮아 CI 자동화보다 단순함을 우선.

### 9. 저장소 정리 + 문서 페이지에 "최종 수정일" 상단 노출 (2026-07-11)
**정리한 파일** (전부 미사용/중복 확인 후 삭제):
- `data/initial_work_guide copy.json` — 첨부파일 필드가 빠진 예전 스냅샷. 코드에서는 `initial_work_guide.json`만 참조, 그냥 남아있던 백업.
- `client/README.md` — Vite 템플릿 기본 안내문 그대로, 프로젝트 내용 없음. 실행법은 루트 `README.md`가 이미 다룸.
- `client/src/assets/{hero.png,react.svg,vite.svg}` — Vite 템플릿 잔여 이미지, `src`/`index.html`/CSS 어디에서도 참조 안 됨.

**문제**: `doc/index.html`·`doc/viewer.html`에 날짜를 적어도, 문서를 다시 고칠 때마다 그 날짜를 손으로 갱신하는 걸 잊어버리면 화면에는 옛날 날짜가 계속 남는다. 게다가 처음엔 그 날짜를 페이지 맨 아래 푸터에만 적어둬서, 스크롤을 끝까지 내리기 전엔 언제 갱신됐는지 보이지도 않았다.

**해결**:
- `doc/viewer.html`: 문서를 `fetch`할 때 응답의 `Last-Modified` 헤더를 그대로 읽어 topbar(맨 위, 문서 열자마자 보이는 위치)에 "🗓 최종 수정 YYYY-MM-DD"로 표시. 파일에 날짜를 직접 적어두지 않으므로 손으로 갱신할 필요가 없음.
- `doc/index.html`: 페이지 자체의 최종 수정일을 `document.lastModified`로 읽어 최상단 eyebrow 영역에 노출(하드코딩 문자열 제거), 하단 푸터의 중복 표기는 삭제.
- 개인 문서 허브(`peter-cho-70.github.io`, 별도 저장소)의 프로젝트 목록도 같은 문제가 있어서 각 카드에 `data-check` 대상 URL(그 프로젝트에서 제일 자주 바뀌는 문서 — NewsControl은 `doc/DEVLOG.md`, MARS/Master는 `doc/STATUS.md`)을 같은 오리진(github.io)으로 `HEAD` 조회해 "최종 갱신"을 카드 앞부분에 자동으로 채워 넣도록 함.

**교훈**: 사람이 손으로 유지보수하는 날짜 문자열은 결국 갱신을 잊어버려 거짓말을 하게 된다. 서버가 이미 알고 있는 값(`Last-Modified` 헤더, `document.lastModified`)을 그대로 읽어 쓰면 갱신을 깜빡할 일이 없다. 또한 "언제 바뀌었는지"처럼 확인하자마자 필요한 정보는 페이지 하단이 아니라 진입 시점에 바로 보이는 위치(topbar, 제목 근처)에 둬야 한다.

### 10. 라이브 배포판(Railway)에 있던 미반영 기능 역설계 병합 + 전광판 표시 규칙 재정의 (2026-07-12)
**배경**: git `main`에는 없는데 실제 Railway 배포(`mbcnewscontrol-production.up.railway.app`)에는 이미 올라가 있는 기능들이 발견됨 — 전광판 1개/2개/3개 이상 표시 그리드, 사용설명서 모달, 저장소 오류 배너, 한줄공지 인라인 수정, 사이드바 빌드 버전 표시. git에는 한 번도 커밋된 적 없는 상태로 배포만 앞서 있었던 것 (다른 세션에서 로컬로 직접 배포했을 가능성이 높음).

**해결**: 배포된 빌드의 압축 JS/CSS 번들을 내려받아 클래스명·상수·로직을 역추적한 뒤 git 소스에 재구현해 병합. 병합하면서 기존 코드의 버그도 같이 고쳤다: `dm-important`/`dm-blink`/`dm-alarm`/`dm-past` 카드 스타일이 "글씨 크게"라는 주석과 반대로 본문 폰트를 오히려 축소시키고 있었음 — Railway 배포판에도 이 버그가 그대로 남아있어서 병합 시점에 함께 제거.

- 전광판 그리드: 표시 중인 공지가 **1개**면 전체화면, **2개**면 좌우 반씩, **3개 이상**이면 기존 2×2(최대 4개) — 구간마다 제목/본문 폰트 크기를 CSS 커스텀 프로퍼티로 스케일링
- 공지 표시 창(display window) 규칙 재정의:
  - 공지일(1일/3일/7일)을 선택 안 하면 예정 시각 기준 **3시간 전 ~ 30분 후**에만 표시(짧게 노출 후 사라짐)
  - 공지일을 선택하면 예정 시각과 무관하게 **등록한 순간(`noticeDayFrom`)부터 선택한 일수 동안** 계속 표시 — 이전에는 "1일"이 항상 암묵적 기본값으로 취급되어 이 구분 자체가 없었음
- 사용설명서(도움말) 모달, 저장소 연결 오류 시 배너, 한줄공지 인라인 수정, 사이드바 빌드 시각 표시(`v{__BUILD_TIME__}`, `vite.config.ts`의 `define`으로 주입) 추가
- 기존 공지 수정 화면에도 게시 기간(공지일) 변경 기능 추가 (이전엔 등록 시에만 가능)

### 12. Railway 배포 지원 추가 (2026-07-12)
**계기**: Vercel 프로덕션 배포가 (원인 미상으로) `BLOCKED` 상태에 걸려 최근 커밋 3개가 며칠째 라이브에 반영되지 않고 있었고, 그 와중에 라이브 사이트조차 Blob 토큰이 연결되어 있지 않아 `/api/guide`가 500을 반환 중이었음. Railway(상시 실행 컨테이너)로 옮기면 서버리스 특유의 `/tmp` 휘발성 문제 자체가 없어지지만, 그래도 데이터/업로드 파일은 기존 Vercel Blob 토큰을 그대로 재사용하기로 함(플랫폼과 무관하게 HTTP로 붙는 API라 재사용 가능, 컨테이너 재배포로 로컬 디스크가 초기화돼도 안전).

- `server/index.js`: 저장소 선택 플래그를 `isServerless`(`process.env.VERCEL`) 하나에서 두 개로 분리 — `isVercel`(정적 클라이언트 서빙·리스닝 여부처럼 "Vercel 서버리스 함수로 실행 중인가"를 뜻하는 것들), `useBlobStorage`(`process.env.BLOB_READ_WRITE_TOKEN` 존재 여부 — 공지/근무 데이터 및 업로드 파일을 Blob에 저장할지). 이렇게 분리해야 Railway처럼 "Vercel은 아니지만 Blob 토큰은 쓰는" 조합이 성립함
- `/api/upload`: 기존엔 업로드 원본·썸네일을 로컬 디스크에만 저장했음(Vercel `/tmp` 휘발성 문제가 여기엔 아예 적용 안 되고 있던 기존 버그). `useBlobStorage`가 켜져 있으면 원본과 썸네일을 Blob에 올리고 로컬 임시 파일은 정리하도록 추가
- `app.listen`에 호스트를 명시적으로 `0.0.0.0`으로 바인딩 — 컨테이너 환경에서 기본 바인딩으로 인한 접속 불가를 예방
- 루트 `package.json`에 `build`/`start` 스크립트와 `engines.node` 추가, `railway.json` 신설(Nixpacks 빌더, 헬스체크 `/`) — 별도 서비스 분리 없이 서버 하나가 클라이언트 빌드까지 서빙하는 기존 구조를 그대로 재사용

**교훈**: "서버리스 환경에서만 영구 스토리지가 필요하다"는 가정은 틀렸다 — 상시 실행 서버라도 컨테이너 재배포 시 로컬 디스크가 초기화되는 플랫폼(Railway 포함 대부분의 PaaS)에서는 똑같이 외부 스토리지가 필요하다. 플랫폼 감지 플래그와 "영구 저장소를 쓸지" 플래그는 처음부터 분리해뒀어야 나중에 다른 플랫폼으로 옮길 때 조건 분기가 꼬이지 않는다.

**추가 결정 (같은 날)**: 막상 Blob 토큰을 다시 쓰려니 Vercel Blob 무료 한도가 이미 꽉 차 있어 유료 전환 없이는 재사용이 불가능했다. Blob 대신 **Railway Volume**(컨테이너에 붙는 영구 디스크)을 쓰도록 전환 — `RAILWAY_VOLUME_MOUNT_PATH`(Volume 연결 시 Railway가 자동 주입)가 있으면 `writableRoot`와 `localDataPath`가 그 경로를 가리키도록 했고, Volume을 처음 붙인 시점엔 파일이 없으므로 리포의 시드 JSON을 한 번만 복사해 초기화하도록 했다. `useBlobStorage` 분기 자체는 그대로 남겨둬서, 이후 Blob 용량 문제가 풀리면 토큰만 다시 넣어도 자동으로 Blob으로 돌아간다.

**교훈**: 외부 유료 스토리지 서비스에 의존하는 설계는 그 서비스의 무료 한도에 발이 묶인다 — 호스팅 플랫폼 자체가 제공하는 디스크(Volume)로 대체 가능한 구조를 처음부터 열어두면, 특정 서비스의 과금 정책에 좌우되지 않고 저장소를 고를 수 있다.

**배포 직전 발견한 문제 (같은 날)**: `railway login`으로 로그인해 프로젝트를 연결해보니, 이 저장소용 Railway 프로젝트(`mbcnewscontrol`)가 **이미 5일 전부터 떠서 실사용 중**이었다 — git에는 그 흔적이 전혀 없는, 이전 세션이 `railway up`으로 직접 올린(커밋 없이) 배포였다(엔트리 9의 "배포 환경이 git보다 앞서 있을 수 있다"가 그대로 재현됨). Volume에는 이미 진짜 공지·카카오 메시지·업로드 이미지 5장이 쌓여 있었는데, `railway volume files list`로 실제 구조를 까보니 데이터는 `<volume>/data/initial_work_guide.json`, 업로드는 `<volume>/server/uploads/...`에 있었다 — 방금 짠 코드는 `<volume>/initial_work_guide.json`, `<volume>/uploads`를 보고 있어서 경로가 한 세그먼트씩 어긋나 있었다. 그대로 배포했으면 기존 데이터를 못 찾고 조용히 시드로 초기화됐을 것(삭제는 아니지만 사용자 입장에서는 "공지가 다 날아간" 것과 동일).

**해결**: `writableRoot`/`localDataPath`가 기존에 이미 쓰이고 있던 그 경로(`<volume>/data/...`, `<volume>/server/uploads/...`)를 그대로 가리키도록 수정. 배포 전 볼륨 파일을 다운로드해서 라이브 API 응답과 byte 단위로 대조 확인 후 `railway up`으로 배포, 배포 후에도 다시 한번 데이터 일치를 확인했다.

**교훈**: "이 배포는 이제 막 설정하는 거니까 빈 상태에서 시작하면 된다"는 가정은 검증 없이 세우면 안 된다 — 특히 여러 세션에 걸쳐 이어지는 프로젝트는 다른(이전) 세션이 이미 무언가를 만들어 놨을 수 있다. 저장소 경로를 새로 정의하기 전에 `railway volume files list` / `railway variables` / 라이브 API 응답으로 실제 상태를 먼저 확인하는 습관이 필요하다.

### 13. 근무자 조회 자동 캡처 공지 + 반복 공지 + 전광판 슬라이드쇼 모드 (2026-07-21)

**계기**: 전광판에 외부 사이트(영어뉴스팀 근무자 조회 도구)를 보여달라는 요청에서 시작. 그 사이트는 `X-Frame-Options: SAMEORIGIN`에 로그인까지 필요해 `<iframe>` 직접 삽입이 불가능함을 확인 — 대신 서버가 헤드리스 브라우저(Playwright)로 매일 로그인해 화면을 스크린샷으로 캡처하고, 그 이미지를 기존 "공지"(KakaoMessage) 데이터에 자동으로 끼워 넣는 방식으로 전환. 전광판 표시 로직은 전혀 건드리지 않고 기존 공지 하나로 취급되게 만든 게 핵심.

- `server/scheduler.js`(신규): 범용 스케줄 엔진. 매분 tick, KST 시각(UTC+9 수동 오프셋, 라이브러리 불필요)과 등록된 시각을 비교, "오늘 이미 실행했는지"는 별도 상태 파일 없이 데이터 자체(같은 sender 태그의 `noticeDayFrom`)로 판단. 실패 시 15분 쿨다운.

- Railway RAILPACK 빌드 환경이 apt-get 접근 가능한 Debian(bookworm) 기반이라 `playwright install --with-deps chromium`이 문제없이 성공함을 실제 배포로 확인 — 계획 단계에서 가장 큰 리스크로 짚었던 부분.

**교훈**: "화면을 그대로 보여줘야 한다"는 요청이 항상 iframe/실시간 임베드를 뜻하는 건 아니다 — 대상이 로그인 필요·프레임 차단 사이트라면, 주기적 스냅샷을 기존 데이터 모델의 항목 하나로 편입시키는 쪽이 기존 로직을 전혀 안 건드리면서 훨씬 안전하다. 또한 "반복"이나 "슬라이드쇼" 같은 새 개념도 완전히 새로운 자료구조를 만들기보다, 기존 필드(`duration`, `timestamp`)의 의미를 그대로 살려 재활용하는 쪽이 회귀 위험이 훨씬 적다.

### 14. 배포 중 로컬 자격증명 파일 실수 유출 + git 저장소 초기화 (2026-07-21)

**문제**: 위 기능에 필요한 로그인 자격증명을 로컬 테스트용 `server/.env.local`(`.gitignore`의 `*.local`로 커밋 제외 대상)에 넣어뒀는데, 이 저장소가 그때까지 **git 저장소 자체가 아니었다**(`git init`을 한 적이 없음). `railway up`은 git과 무관하게 현재 디렉터리를 그대로 업로드하는데, `.gitignore` 기반 제외 로직은 실제 git 저장소가 있어야 정상 동작하는 것으로 보임 — 그 결과 첫 배포에 `server/.env.local`(평문 로그인 자격증명 포함)이 그대로 실려 올라갔다. 배포 후 런타임 로그의 `injected env (2) from .env.local`로 뒤늦게 발견.

**해결**: (1) 자격증명을 정식 Railway 환경변수(`railway variables --set`)로 등록. (2) `git init` + 첫 커밋으로 이 저장소를 실제 git 저장소로 전환 — 이후 `.gitignore`가 배포 업로드에도 제대로 적용됨을 확인(`server/.env.local`, `server/uploads`, `node_modules` 등이 스테이징에서 정상 제외됨). (3) 재배포해서 컨테이너에서 해당 파일이 완전히 빠졌는지(`injected env (0) from .env.local`) 확인.

**교훈**: `railway up`(혹은 유사한 "현재 디렉터리를 그대로 업로드"하는 배포 CLI)은 git 저장소가 실제로 존재해야 `.gitignore`가 제 역할을 한다 — 이 프로젝트처럼 git 없이 배포만 해온 저장소는 `*.local`/`.env*` 같은 민감 파일이 조용히 배포 번들에 실릴 수 있다. 배포 전에 반드시 `git status`/`.git` 존재 여부를 먼저 확인하고, 없다면 최소한 `git init` + 첫 커밋을 해서 gitignore가 실제로 작동하는 상태를 만든 다음 배포할 것.

### 15. 전광판 제목 설정값화 + 두 번째 조직(가정용) 배포 (2026-07-22)

**계기**: 같은 기능을 다른 조직(5인 이하, 텍스트+사진 위주 — 가정용 공지판)에도 쓰고 싶다는 요청. 지금 구조는 "배포 하나 = 데이터 하나"라 같은 인스턴스를 공유하면 두 조직 공지가 섞이므로, 같은 코드베이스를 Railway에 한 번 더(별도 프로젝트/서비스/Volume) 배포하는 방식을 택함.

- 소스에 하드코딩된 조직명은 `client/src/App.tsx`의 `?display` 헤더 "보도기술팀 공지사항" 문구 하나뿐이었음 — 이를 `displayTitle` 설정값(기존 `alertSettings`/`captureSettings`/`slideshowSettings`와 동일한 패턴, 기본값은 기존 문구 그대로)으로 전환해서 코드 수정 없이 배포마다 다른 제목을 쓸 수 있게 함.
- Railway에 `railway up --new`로 새 프로젝트("우리집공지판") 생성, `railway volume add`로 전용 Volume 연결(연결 안 하면 컨테이너 재배포마다 데이터가 초기화되므로 필수), `railway domain`으로 공개 URL 발급.
- 새 배포는 최초 부팅 시 리포의 시드 데이터(MBC 방송 전용 역할/프로그램)로 자동 초기화되므로, 배포 직후 API로 `roles`/`programs`/`periodic_events`를 빈 배열로, `displayTitle`을 "우리집 공지사항"으로 덮어써서 깨끗한 상태로 전환.
- Railway 사용량 확인 결과 이 워크스페이스는 완전 무료가 아니라 사용량 기반 과금 중(당시 15일간 약 $0.45) — 새 서비스 하나 추가해도 5인 이하·캡처 기능 미사용 기준으로는 증가폭이 크지 않을 것으로 예상.

**교훈**: 여러 조직/용도로 재사용할 가능성이 있는 앱은 "하드코딩된 문구가 딱 하나뿐"이어도 그 하나 때문에 배포마다 소스를 갈라치기하게 될 수 있다 — 발견 즉시 기존 설정 패턴을 재사용해 설정값으로 빼두면, 코드 한 벌로 여러 배포를 영구히 재사용할 수 있다. 또한 새 배포 환경(Railway Volume)은 항상 시드 데이터로 처음 초기화된다는 점을 감안해, 배포 직후 "실제 조직 데이터로 덮어쓰기"를 표준 절차로 삼을 것.

### 16. 좌측 메뉴 이름 변경 + 표시/숨김 (2026-07-22)

**계기**: 좌측 사이드바 메뉴(대시보드/업무 가이드/뉴스별 TD 업무/행사 아카이브/점검 아카이브/공지 피드/설정 + 커스텀 탭) 이름을 바꾸고 켜고 끌 수 있게 해달라는 요청, 두 배포(MBC/우리집) 모두에 적용.

- `client/src/App.tsx`: 고정 메뉴 6개는 라벨이 JSX에 하드코딩돼 있었고 커스텀 탭은 생성 시에만 이름을 지정할 수 있었음(수정 불가, 삭제만 가능) — `activeTab` 라우팅 키나 각 탭의 본문 렌더링은 그대로 두고, **사이드바에 뭐라고 표시할지/보일지만** `menuSettings`(고정 메뉴용 라벨·숨김 오버레이) + `CustomTab.hidden`(커스텀 탭용)으로 데이터화.
- 고정 메뉴 목록 자체(id/기본라벨/아이콘)는 아이콘이 React 컴포넌트라 데이터로 못 빼서 `FIXED_NAV_ITEMS` 클라이언트 상수로 유지, `menuSettings`는 그 위에 라벨/숨김만 덮어씀.
- **안전장치**: "설정" 메뉴는 라벨 변경은 되지만 숨김 토글 자체를 제공하지 않음 — 실수로 다 숨겨도 설정으로 돌아가 되돌릴 진입점을 항상 보장. 지금 보고 있는 탭이 숨겨지면 대시보드로, 대시보드마저 숨겨졌으면 설정으로 자동 대피하는 `useEffect`도 추가.
- 두 배포 다 재배포해서 확인 — 기존 배포엔 `menuSettings`가 없으므로(빈 값으로 처리) 라벨/표시 상태가 지금까지와 동일하게 유지됨(회귀 없음)을 실제로 확인.

**교훈**: "메뉴를 숨길 수 있게 해달라"는 요청은 "관리자가 실수로 자기 자신을 잠글 수 있는" 기능이기도 하다 — 이런 기능을 만들 때는 항상 되돌릴 수 있는 최소 진입점(여기선 "설정" 메뉴)을 하나는 숨김 대상에서 제외해 안전망으로 남겨둬야 한다.

### 17. 우리집 배포 주소를 간단하게 변경 (2026-07-22)

**계기**: 우리집공지판 배포의 Railway 자동 생성 주소(`positive-mercy-production-0428.up.railway.app`)가 너무 복잡해서 라즈베리파이 키오스크 등에서 쓰기 불편하다는 요청.

- Railway CLI의 `railway domain update <기존주소> --domain <새 호스트 이름>`으로 Railway가 제공하는 서비스 도메인의 호스트 이름(`up.railway.app` 앞부분)만 원하는 문자열로 즉시 변경 가능함을 확인. 별도 DNS 설정이나 소유 도메인 없이도 바로 적용됨(자체 도메인을 쓰려면 `railway domain <도메인>`으로 별도 연결 + DNS 레코드 추가가 필요하지만, 이번엔 그 정도까지는 필요 없었음).
- `https://ourhome.up.railway.app`로 변경, 정상 응답(`/`, `/?display`) 확인.

**교훈**: Railway가 자동 생성한 무작위 서비스 주소는 CLI 한 줄(`railway domain update ... --domain ...`)로 바로 바꿀 수 있다 — 커스텀 도메인 연결처럼 DNS 설정이 필요한 큰 작업이 아니다. 다만 주소를 바꾸면 이미 그 주소를 하드코딩해둔 클라이언트(키오스크 브라우저 등)도 같이 갱신해야 한다.

## 지금까지의 핵심 교훈 정리

1. **Vercel + 모노레포 정적 빌드**: `@vercel/static-build`의 산출물 경로는 직관과 다르다. 새 정적 자산을 추가하면 `vercel.json` 라우트도 같이 점검할 것.
2. **네이티브 모듈은 지연 로딩**: 서버리스 런타임에서 `sharp` 같은 네이티브 의존성은 모듈 최상단에서 요구하면 전체 함수가 죽는다. 실제로 쓰는 시점에 로드하고 실패를 격리할 것.
3. **서버리스 = 상태 없음**: `/tmp` 등 로컬 파일시스템에 의존하는 쓰기는 콜드스타트마다 사라진다. 영속 데이터는 반드시 외부 스토리지(Blob 등)로.
4. **일시적 에러 ≠ 데이터 없음**: 외부 스토리지 읽기 실패 시 무조건 시드값으로 폴백하면 실데이터가 유실될 수 있다. "존재하지 않음"과 "읽기 실패"를 구분해서 처리.
5. **날짜는 항상 로컬 타임존 기준으로**: `toISOString()`은 UTC라서 한국 새벽 시간대에 날짜가 하루 밀리는 버그를 만든다.
6. **정적 서버의 서빙 루트 = 상대 경로가 닿을 수 있는 범위**: `http.server --directory doc`처럼 하위 폴더를 루트로 띄우면 `../`로 그 바깥 파일을 못 읽는다. 문서 뷰어가 저장소 루트의 README/PRD를 상대 경로로 참조한다면, 서버도 저장소 루트에서 띄워야 한다.
7. **private 저장소는 GitHub Pages를 바로 못 켠다(무료 플랜 기준)**: 문서만 공개하고 싶으면 코드 저장소는 private으로 두고, 문서만 미러한 별도 public `-docs` 저장소를 만들어 그쪽에서 Pages를 켠다.
8. **날짜는 사람이 아니라 서버가 기억하게 한다**: 문서에 "최종 수정일"을 손으로 적으면 다음 수정 때 빠뜨려서 거짓 정보가 남는다. `Last-Modified` 헤더나 `document.lastModified`처럼 서버가 이미 아는 값을 읽어 표시하고, 그 값은 스크롤 없이 바로 보이는 위치(상단)에 둔다.
9. **배포 환경이 git보다 앞서 있을 수 있다**: 로컬에서 직접 배포하거나 다른 세션에서 작업하면, 소스가 커밋 하나 없이 배포만 되어 있을 수 있다. 압축된 프로덕션 JS/CSS 번들을 받아 클래스명·상수·로직을 리버스 엔지니어링하면 실제로 뭐가 다른지 정확히 확인해서 병합할 수 있다.
10. **로컬 개발 서버는 여러 세션이 동시에 쓸 수 있다**: 테스트 데이터를 정리한다고 스냅샷을 통째로 복원하면, 그사이 실사용자(또는 다른 세션)가 추가한 데이터까지 같이 날아간다. API 응답을 diff로 먼저 확인해 내가 만든 항목만 골라 지울 것.
11. **새 배포 플랫폼도 "빈 상태"라고 가정하지 말 것**: 다른 세션이 이미 그 플랫폼에 커밋 없이(`railway up` 등 CLI 직접 업로드로) 배포해뒀을 수 있다. 저장소 경로를 새로 정의하기 전에 `railway volume files list` / `vercel env ls` 같은 명령으로 실제 인프라 상태부터 확인할 것 — 안 그러면 실데이터 경로와 어긋난 코드를 배포해 조용히 초기화시킬 수 있다.
12. **git 저장소가 없으면 `.gitignore`가 배포에 적용 안 될 수 있다**: `railway up` 같은 "현재 디렉터리 그대로 업로드" 방식 배포 CLI는 `.gitignore` 제외 로직이 실제 git 저장소 존재를 전제로 하는 것으로 보인다. git이 없는 상태로 배포하면 `*.local`/`.env*` 같은 민감 파일이 조용히 번들에 실릴 수 있으니, 배포 전에 `git status`로 저장소 여부부터 확인하고 없다면 최소 `git init` + 첫 커밋 후 배포할 것.
